- **[CRITICAL] devground-setup configura lint-staged como string en package.json — formato que lint-staged >=16 rechaza en runtime — y a la vez instala el pre-commit que lo ejecuta: todos los commits quedan bloqueados tras el quickstart** (`qa-happy`)
  - `packages/devground/setup.js:53-59, 174`
  - Evidencia: setup.js:55 hace `pkg['lint-staged'] = '@devground/lint-staged-config'` (string en package.json) y luego escribe `.husky/pre-commit` con `pnpm exec lint-staged` (línea 174). lint-staged 16.4.0 (validateConfig.js:15-16) lanza `ConfigFormatError: 'Configuration should be an object or a function'` para cualquier config que no sea objeto/función — un string NO se resuelve como módulo. El propio monorepo lo sabe: packages/cli/src/installers/lint-staged.ts comenta "lint-staged does NOT resolve a string — it rejects it at runtime" y por eso el CLI escribe lint-staged.config.cjs. El mismo consejo roto vive en packages/lint-staged-config/README.md:13-19 (sección Usage). El peer del agregador exige lint-staged >= 16.0.0.
  - Por qué importa: Es el flujo estrella del README raíz ("Dos comandos y listo"): tras `npx devground-setup`, el primer `git commit` dispara el hook, lint-staged lee la clave string y aborta → todos los commits bloqueados hasta arreglar la config a mano. El bug ya fue diagnosticado y corregido en devground-init (escribe .cjs, con test de regresión en installers.test.ts:138-150), pero setup.js y el README de @devground/lint-staged-config siguen instalando/promoviendo el patrón roto. Hallazgo HIGH histórico de la skill aún sin cerrar en estos dos lugares. Fix: replicar el patrón del CLI (lint-staged.config.cjs) y corregir el README del preset.

1) Los comandos `report` y `diff` son SOLO-LECTURA: `report` hace `log(renderReport(loadSnapshot(...)))` y `diff` hace `log(renderDiff(...))` — ambos solo imprimen a stdout, no escriben nada. Un snapshot válido siempre está bien formado porque lo escribió la herramienta.

3) Esto es análogo a las supresiones ya establecidas en la skill de auditoría (manifest `as { version: string }` y `response.tools as string[]`): casts sobre artefactos internos controlados que "no generan daño en este flujo". El único escenario de fallo real es que el usuario pase el archivo equivocado, que ya termina en exit(1) con mensaje.

El residuo real es únicamente la OPACIDAD del mensaje de error en un escenario de error-de-usuario con un artefacto auto-generado — a lo sumo un nicety de diagnóstico de severidad low/cosmética, no un defecto de correctitud medium.

1) La exclusión es intencional y documentada, no un bug. El `case "$CMD" in *';'*|*'&'*|*'|'*|*$'\n'*) ;;` es la implementación explícita de ese requisito. El allowlist inline es solo una conveniencia estrecha para trabajo ya revisado; su no-aplicación a comandos complejos degrada a "delega en vez de inline", que es exactamente el comportamiento nuclear de la regla, no una rotura. Es fail-closed (el propio hallazgo lo admite: "no inseguro").

Además, "el caso más común" es discutible: los conventional commits de subject única (`feat: x`, `fix: y`) — que son lo que el CLAUDE.md del usuario manda ("Use conventional commits only", "Never add Co-Authored-By", lo que elimina la razón principal del heredoc) — SÍ pasan el allowlist. En suma: comportamiento intencional + documentado + fail-closed + con fallback de delegación que completa la operación. No es un defecto de correctitud ni un contrato incumplido.
