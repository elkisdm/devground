Razón: El hallazgo es un falso positivo causado por un grep mal acotado a `packages/cli/src/`. Ese setup.js, en las líneas 48-51, lee el `package.json` del proyecto destino y escribe `pkg.scripts.prepare = 'husky'` (el patrón estándar de husky v9). El propio hallazgo reconoce que "el enganche de husky depende del CLI externo npx devground-husky", pero concluye erróneamente que eso significa que no hay edición de package.json — cuando ese mismo CLI externo es exactamente lo que escribe el script `prepare`. - ~~detectFramework / readPackageJson no validan la forma del package.json externo y castean con `as` en la frontera no confiable~~ (`aud-types`, medium) — refutado 2/3. Razón: El hallazgo es un falso positivo en su núcleo argumental: invoca dos ADRs cuyo alcance explícito NO cubre este flujo.

1) ADR-0011 ("Prohibido `any` en fronteras externas (DB/API)") aplica, por su propio texto (línea 6), a "proyectos que tocan DB/API externas (Supabase, ORMs, clientes HTTP)" y su enforcement (líneas 44-46) es `@typescript-eslint/no-explicit-any`. El antipatrón que condena es `as any` / `as unknown as Foo`. El código aquí usa `as Record<string, unknown>` y `as Record<string, string>` — NO son `any`, no los marca `no-explicit-any`, y la frontera es la lectura de un archivo local (package.json del directorio destino del propio usuario), no DB/API. El finding estira ADR-0011 fuera de su alcance declarado.

2) ADR-0009 ("Validación de entrada en toda ruta API + firma en webhooks") aplica (línea 6) a "proyectos con rutas API ... y receptores de webhooks"; su modelo de amenaza es payloads de red no confiables (`req.json()`), HMAC y timing attacks. Un package.json leído del disco local no es ruta API ni webhook. No aplica.

3) El propio finding concede que el impacto de seguridad es nulo hoy: los valores detectados (framework/hasTypeScript/packageManager) solo fluyen a enums y a un mensaje al usuario, nunca a `run()`. `addDevDependency` arma el comando con nombres de paquete hardcodeados, no con datos parseados. El "deja la puerta abierta a un refactor futuro" es especulativo, no un defecto presente.

4) Salvaguarda real existente: el JSON malformado lanza excepción y está testeado (package-json.test.ts:43-47 "throws when package.json is malformed JSON"). TypeScript strict (ADR-0004) más la suite de tests definen el contrato.

5) El sub-hallazgo de detectFramework (solo mira dependencies+devDependencies) describe comportamiento real, pero detect-stack.test.ts:40-44 codifica explícitamente `unknown` como resultado INTENCIONADO para un paquete sin deps regulares, y el README (línea 23) solo promete detectar Next.js/React/Node. Un paquete que exponga un framework únicamente vía peerDependencies es un edge case fuera de los requisitos documentados — decisión de diseño, no defecto.

El cast `as` es a lo sumo un code-smell de robustez (validar `typeof === 'object' && !Array.isArray` sería más limpio), pero etiquetarlo como violación de ADR-0011/0009 e inflar una decisión de diseño documentada a defecto es incorrecto. Ante la duda razonable y con el alcance de los ADRs en mano, se refuta.
- ~~Los errores de comandos npx/instalación pierden el detalle cuando la herramienta escribe diagnósticos en stdout~~ (`qa-errors`, medium) — refutado 2/3. Razón: El mecanismo descrito es real pero el impacto NO aplica a este flujo. Verifiqué que run() (exec.ts:8) usa stdio:'pipe' y descarta e.stdout, e index.ts:111-112 solo imprime err.message. Sin embargo, TODOS los comandos que devground-init ejecuta realmente enrutan sus diagnósticos a stderr, que execSync con stdio:'pipe' SÍ incorpora a e.message. Comprobado empíricamente: (1) `npm install -D <pkg-inexistente>` → stderr_len=537, stdout_len=0, y e.message contiene "npm error code E404 ... could not be found"; (2) `npx <bin-inexistente>` → stderr_len=515, stdout_len=0, detalle presente en e.message. Las únicas llamadas a run() en el flujo son `npm/pnpm/yarn add -D` (package-json.ts:43) y `npx devground-{husky,agents,architecture}` (husky.ts:10, agents-md.ts:10, architecture-guide.ts:10). Los scripts first-party devground-* usan @devground/logger cuyo error() escribe a console.error (logger/src/index.ts:30), es decir stderr, por lo que también quedan en e.message. El escenario dañino del hallazgo (fallo que escribe el detalle SOLO en stdout) no se materializa para ninguno de estos comandos: npm/pnpm/yarn/npx emiten errores por stderr por convención. La afirmación "común en varios package managers y scripts npx" queda contradicha por el comportamiento real. El propio hallazgo reconoce que con stdio:'pipe' stderr va a e.message. Por tanto la causa raíz NO se pierde en las rutas de fallo realistas de este flujo; es un caso teórico que no aplica.

- ~~run() ejecuta comandos vía execSync con string interpolado (shell habilitado), patrón inyectable por diseño, y addDevDependency es un sink latente~~ (`aud-security`, medium) — refutado 2/3. Razón: El código es factualmente correcto tal como se describe: exec.ts:7-9 usa `execSync(cmd, ...)` con un string, y package-json.ts:42-43 concatena `packages.join(' ')` en el comando. Pero el hallazgo NO aplica como vulnerabilidad real en el flujo devground-init por tres razones verificadas:

1. NINGÚN dato no confiable llega al sink. Revisé TODOS los callers (grep en packages/cli/src):
   - addDevDependency: prettier.ts, eslint.ts, tsconfig.ts, commitlint.ts, lint-staged.ts, husky.ts, agents-md.ts, architecture-guide.ts — todos pasan literales hardcodeados ('@devground/prettier-config', 'eslint', 'typescript', etc.). eslint.ts:12-14 arma un array, pero solo con literales seleccionados por `stack.framework === 'nextjs'` (enum cerrado).

   El nombre de paquete jamás proviene de package.json, args de CLI ni input del usuario. El propio hallazgo admite: "Hoy NO es explotable en devground-init".

2. La cita a ADR-0009 es incorrecta. ADR-0009 (docs/adr/0009-validacion-entrada-webhooks.md) está explícitamente acotado a "rutas API (Next.js Route Handlers, serverless functions) y receptores de webhooks": validación de payload con Zod y verificación de firma HMAC. NO trata sobre construcción de comandos shell en un CLI. Listé los 12 ADRs (0001-0012) y NINGUNO gobierna ejecución de shell / inyección de comandos en el CLI. La "frontera única y segura" de ADR-0009 es la frontera entrada-no-confiable→lógica en HTTP, no el sink de execSync.

3. El argumento "el contrato no ofrece barrera" / "sink latente" describe un riesgo futuro hipotético que requiere un caller que no existe, no una vulnerabilidad presente. Es una sugerencia de hardening de calidad de código, no un hallazgo de seguridad en este flujo.

Bajo la lente de refutación (sin dato no confiable que alcance el sink, ADR citado no aplica, no-explotabilidad auto-admitida), es un falso positivo como hallazgo de severidad medium para devground-init.

- ~~La selección del prompt interactivo (entrada de usuario) se castea a string[] sin validar~~ (`aud-types`, low) — refutado 3/3. Razón: El cast existe (index.ts:96 `const selectedValues = response.tools as string[];`) y `response.tools` efectivamente es `any` según @types/prompts (`Answers<T> = { [id in T]: any }`). Hasta ahí el hecho técnico es correcto. Pero el hallazgo se apoya en una premisa FALSA sobre la rúbrica: afirma que "ADR-0011 cubre explícitamente 'entrada de usuario' como frontera externa". Leí el ADR completo (docs/adr/0011-prohibido-any-fronteras-externas.md): su título, "Aplica a" y todo el cuerpo lo limitan a "fronteras externas (DB/API)" — "proyectos que tocan DB/API externas (Supabase, ORMs, clientes HTTP)". El Contexto, la Decisión y las Referencias tratan EXCLUSIVAMENTE del antipatrón `supabase as any` en el límite con la base de datos. Un grep del ADR por "entrada de usuario/user input/stdin/interactiv/prompt" no devuelve nada. El ADR NO cubre input interactivo de CLI; el hallazgo le atribuye un alcance inventado. Además, el enforcement del ADR es `@typescript-eslint/no-explicit-any` (eslint-config/next.mjs:42 'warn'), y la línea usa `as string[]`, NO `as any` ni `as unknown as Foo`: no introduce `any` y no la marcaría esa regla. En cuanto a correctitud real: el guard previo (líneas 91-94) ya descarta vacío/undefined, los valores provienen de choices construidos por el propio código (`ALL_INSTALLERS.map(i => ({value: i.value}))`), y el único uso posterior es `selectedValues.includes(i.value)`, inofensivo aunque el shape no fuera string[]. No hay defecto de comportamiento; el supuesto incumplimiento del ADR no aplica.
- ~~El parseo del propio manifest se castea a { version: string } sin validar~~ (`aud-types`, low) — refutado 3/3. Razón: El código en index.ts:39-41 coincide literalmente con la evidencia: JSON.parse(...) as { version: string }. La afirmación mecánica ("se castea sin validar") es cierta a nivel sintáctico, PERO la atribución a ADR-0011 es incorrecta y constituye el fallo del hallazgo. ADR-0011 ("Prohibido any en fronteras externas DB/API") está estrictamente acotado a: (1) el operador any/as any/as unknown as Foo, cuyo enforcement es @typescript-eslint/no-explicit-any (líneas 1, 22-46), y (2) la frontera con DB/API externas — Supabase, ORMs, clientes HTTP (línea 6: "todos los proyectos que tocan DB/API externas"; línea 10: "el límite con la base de datos"). El cast del manifest NO encaja en ninguno de los dos criterios: (a) no usa any sino un tipo concreto { version: string }, por lo que no_explicit-any ni siquiera lo marcaría; (b) no es una frontera externa no confiable: lee el package.json propio EMPAQUETADO del paquete en build-time (comentario en líneas 36-38: "Single source of truth... the package manifest"), que es lo opuesto a una entrada externa. El propio hallazgo lo reconoce ("lee su propio manifest empaquetado, no entrada de usuario", severidad low). Un package.json válido siempre tiene version (npm lo exige para publicar/instalar); un manifest corrupto fallaría antes de llegar aquí. El patrón es intrínsecamente seguro y el ADR citado como rúbrica no gobierna este caso, por lo que es un falso positivo.
- ~~El orquestador index.ts (preset routing, tally de fallos, process.exit y agregación de errores) no tiene NINGÚN test~~ (`aud-tests`, high) — refutado 2/3. Razón: El hecho técnico crudo es cierto (no existe index.test.ts; el path `catch { failures++ }` y la decisión `if (failures > 0) process.exit(1)` en index.ts:105-128 no tienen cobertura, y el fake `run` de installers.test.ts:69-72 nunca lanza). ADR-0012 define "ruta crítica" con una taxonomía CERRADA de exactamente tres categorías (líneas 20-24): (1) Dinero —precios, montos, impuestos, comisiones, cobros, reembolsos—, (2) Leads/datos de conversión, (3) Auth/autorización. Y se autolimita en su alcance (línea 6: "Aplica a: cualquier proyecto del repo con lógica de dinero, captación de leads o autenticación"). El orquestador del CLI (ruteo de presets `opts.yes || opts.preset === 'full'`, conteo de fallos, `process.exit(1)`) no toca dinero, ni leads, ni auth: NO cae en ninguna de las tres categorías, por lo que el requisito "debe tener tests unitarios antes de desplegar" (línea 28) NO le aplica. El ADR incluso rechaza explícitamente exigir cobertura amplia: línea 14 "No se trata de exigir 100% de cobertura en todo", línea 53 "Mejor cobertura profunda en lo crítico que ancha y superficial", descartando umbrales globales que generan "tests de relleno sin valor". El origen del ADR (línea 60) confirma el scope: un repo que tocaba DINERO con cero tests. El hallazgo inventa una noción de "ruta crítica de un scaffolder" que el ADR no contiene y la usa como rúbrica obligatoria; bajo la propia convención del proyecto, esa obligación no existe para este código. Es una mejora de robustez deseable, pero no un incumplimiento de la rúbrica citada ni un hallazgo "high".
- ~~package-json.ts: addDevDependency y getInstallCmd (construcción del comando real de shell) sin test~~ (`aud-tests`, low) — refutado 3/3. Razón: La descripción técnica del hallazgo es FACTUALMENTE CORRECTA: package-json.ts:27-44 contiene getInstallCmd (switch que mapea pnpm/yarn/npm a 'pnpm add -D' / 'yarn add -D' / 'npm install -D') y addDevDependency, que arma `${getInstallCmd(pm)} ${packages.join(' ')}` y lo pasa a run() (exec.ts ejecuta execSync). Verifiqué que package-json.test.ts solo importa readPackageJson y writePackageJson (línea 5), e installers.test.ts usa un fake de addDevDependency que solo registra llamadas (líneas 57-73), nunca ejercitando la construcción real del string. No existe ningún test del string generado. Sin embargo, el hallazgo es FALSO POSITIVO como issue accionable bajo la rúbrica de auditoría del propio proyecto. ADR-0012 (docs/adr/0012-tests-rutas-criticas.md) define "ruta crítica" de forma cerrada y exclusiva: (1) dinero, (2) leads/datos de conversión, (3) auth/autorización (líneas 20-24), y exige tests SOLO para esas categorías. El ADR rechaza explícitamente exigir cobertura amplia: "No se trata de exigir 100% de cobertura en todo" (línea 14) y descarta la alternativa de cobertura global alta porque "Produce tests de relleno (getters, mappers triviales) que dan falsa confianza y poco valor" (línea 53). El propio hallazgo admite "No es money/leads/auth". getInstallCmd es exactamente un mapper trivial de 3 casos sobre un union cerrado PackageManager (el switch exhaustivo en TypeScript garantiza retorno de string para todo pm válido), y la propagación del package manager YA está cubierta end-to-end en installers.test.ts (líneas 289-298 verifican que pm llega a addDevDependency, y cada test asevera el array packages exacto). Esto es una herramienta CLI de scaffolding con CERO superficie de dinero/leads/auth. El hallazgo describe correctamente el código pero reporta como defecto la ausencia de test en un mapper trivial no-crítico, lo que contradice directamente la rúbrica del proyecto (ADR-0012). Ante la duda razonable y dado que el ADR excluye este caso exacto, se refuta.
